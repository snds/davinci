# Breakpoints & Layout

Davinci provides named breakpoint tokens and responsive layout utilities to ensure consistent adaptive behavior across all surfaces. All breakpoint values are exposed as CSS custom properties so they can be referenced in both CSS and JavaScript.

## Breakpoints

Five named breakpoints define the responsive grid. They align with Tailwind's default scale so utility classes and token-based code stay in sync.

### Usage in CSS

Breakpoint tokens are CSS custom properties, not native `@media` conditions — so they cannot be used directly inside `@media`. Use them in JavaScript or alongside named constants:

```css
/* ✅ Use the token value in JS */
const isMobile = window.innerWidth < getComputedStyle(document.documentElement)
  .getPropertyValue('--bp-sm').trim();

/* ✅ Use in media queries by value */
@media (min-width: 640px) { /* --bp-sm */ }
@media (min-width: 1024px) { /* --bp-lg */ }

/* ❌ This does NOT work — CSS variables can't be used in @media */
@media (min-width: var(--bp-sm)) { }
```

### Tailwind alignment

All breakpoint names map to Tailwind v4 prefix shorthands (`sm:`, `md:`, `lg:`, `xl:`, `2xl:`). Keep Tailwind utility classes and token-based breakpoints in sync — never introduce a custom viewport that sits between named steps.

### Page inset pattern

Every page-level wrapper should use `--page-inset-x` as its horizontal padding. This creates a consistent safe zone across all viewports without requiring individual `@media` overrides in each component:

```css
.page-wrapper {
  padding-left:  var(--page-inset-x);
  padding-right: var(--page-inset-x);
}
```

```jsx
/* In React inline styles */
<div style={{ padding: '0 var(--page-inset-x)' }}>
  {/* content */}
</div>
```

### Container widths

Constrain content width with `--container-doc` or `--container-max`, then center with `margin: 0 auto`:

```css
.doc-content {
  max-width: var(--container-doc);
  margin: 0 auto;
  padding: 0 var(--page-inset-x);
}

.app-shell {
  max-width: var(--container-max);
  margin: 0 auto;
}
```

## Responsive layout guide

| Viewport | Columns | Rails | Page inset |
|---|---|---|---|
| `< --bp-sm` | 1 | Hidden | 20px |
| `--bp-sm → --bp-lg` | 1–2 | Left only | 24px |
| `≥ --bp-lg` | 2–3 | Both visible | 32px |

### Pattern-specific breakpoints

| Pattern | Hide right rail | Collapse left rail | Single column |
|---|---|---|---|
| Feed | `< --bp-xl` | `< --bp-lg` | `< --bp-md` |
| Profile | `< --bp-xl` | `< --bp-lg` | `< --bp-md` |
| Rails | `< --bp-xl` | `< --bp-lg` | `< --bp-md` |

## Dark mode integration

Davinci's theme system uses `data-theme="dark"` on the `<html>` element as the authoritative signal for dark mode. The Tailwind dark variant supports both `data-theme="dark"` and the `.dark` class so the system works in Docusaurus, standalone React apps, and any host using shadcn's default class-based approach:

```css
/* In packages/ui/src/styles/globals.css */
@custom-variant dark (&:is(.dark *, [data-theme="dark"] *));
```

**For host applications using Docusaurus:** the `Root` swizzle component (`src/theme/Root/index.jsx`) uses a `MutationObserver` to keep `.dark` in sync with `data-theme`, ensuring any embedded shadcn components that check the class directly continue to work.

**For standalone React apps:** add `dark` to `<html>` when the user selects dark mode, or set `data-theme="dark"` — either works.
